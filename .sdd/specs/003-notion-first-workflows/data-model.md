# Data Model: Notion-First GTM Workflows

## Existing Entities

### Target Account

Represents the public sector account being pursued.

**Current Notion database**: `B2G Target Accounts`

**Relevant relations**:

- Has many SECOP Research records.
- Has many Opportunities.
- Has many GTM Outputs.
- May have an Owner.
- May belong to an ICP.

### SECOP Research Record

Represents one normalized evidence record from SECOP or related public procurement datasets.

**Current Notion database**: `B2G SECOP Research`

**Stable identity**:

- Source Platform
- Source Record ID

**Relevant relations**:

- Belongs to one Target Account when matched.
- May support one or more Opportunities.
- May support one or more GTM Outputs.

### Opportunity

Represents a potential commercial pursuit derived from account context and procurement evidence.

**Current Notion database**: `B2G Opportunities`

**Relevant relations**:

- Belongs to a Target Account.
- References supporting SECOP Research records.
- Has many GTM Outputs.

### GTM Output

Represents a generated sales artifact such as outreach, meeting prep, proposal brief, or business case.

**Current Notion database**: `B2G GTM Outputs`

**Current properties**:

- Title
- Type
- Content
- Source Summary
- Approval Status
- Target Account
- Opportunity
- Research Records
- Created At
- Updated At

**Proposed additional properties**:

- Output Key: rich text; stable dedupe key.
- Generated By: select or rich text; e.g. `agent`.
- Source Evidence Hash: rich text; hash of supporting evidence IDs/content used.
- Stage: select; e.g. `draft`, `ready_for_ae`, `sent`, `used_in_meeting`, `archived`.
- Owner: relation to `B2G Owners`, if the output is assigned.
- Channel: select; e.g. `email`, `linkedin`, `meeting`, `proposal`, `internal`.

**Body content**:

- Rendered output artifact as Notion blocks.
- Optional "Evidence used" section with links/relations to source records.
- Optional "Next action" section for AE workflow.

## Proposed New Entity

### Research Run

Represents run-level metadata for one research execution.

**Proposed Notion database**: `B2G Research Runs`

**Priority**: Optional for first implementation; recommended before recurring/automated research.

**Properties**:

- Name: title.
- Run Key: rich text; stable run identifier or deterministic hash.
- Status: select; `completed`, `partial`, `failed`.
- Started At: date.
- Completed At: date.
- Source Mode: select; `real_source`, `sample_data`, `manual_import`.
- Dataset Count: number.
- Raw Record Count: number.
- Deduped Record Count: number.
- Notes: rich text.
- Target Account: relation to `B2G Target Accounts`.
- Research Records: relation to `B2G SECOP Research`.

**Rationale**:

Without Research Run pages, run-level audit metadata stays local or gets repeated across records. This may be acceptable for MVP, but a dedicated database gives users a clearer audit trail inside Notion.

## Dedupe Keys

### SECOP Research Record

```text
secop:<source_platform>:<source_record_id>
```

### GTM Output

```text
gtm-output:<type>:<target_account_key>:<opportunity_key>:<source_evidence_hash>
```

If an opportunity page ID exists, use it as the opportunity key. If not, derive from title, target account, source URL, and estimated value.

### Research Run

```text
research-run:<input_hash>:<started_date_or_external_run_key>
```

If research becomes scheduled, the run key should include the schedule ID and execution timestamp.

## State Transitions

### GTM Output Approval Status

```text
draft -> needs_review -> approved
draft -> rejected
approved -> archived
needs_review -> draft
```

Agents may create or update draft outputs. Humans should move outputs to approved/rejected.

### GTM Output Stage

```text
draft -> ready_for_ae -> sent
draft -> ready_for_ae -> used_in_meeting
any -> archived
```

Stage is operational; Approval Status is governance/review.

## Migration Mapping

Migration is one-way from local artifacts into Notion. After an artifact is imported, future workflows must read the Notion page or database record, not the original local file.

### Local output markdown to GTM Output

- File name or explicit argument -> Type.
- First markdown heading -> Title.
- Full markdown -> page body.
- Source JSON path, when available -> Source Summary.
- Linked opportunity/account -> relations when resolvable.
- Otherwise -> create unlinked draft with Source Summary explaining missing relations.
- Once imported -> Notion GTM Output page becomes the only reusable business object.

### Local SECOP run to Notion

- `secop-research.jsonl` records -> `B2G SECOP Research`.
- `manifest.json` -> `B2G Research Runs` if that database is adopted.
- Run ID -> internal audit field or Research Run relation, not primary user-facing result.
- Once imported -> Notion records become the only reusable evidence input.

## Local Artifact Policy

Local artifacts are not part of the GTM data model. They are allowed only in narrow cases:

- Test fixtures used by automated tests.
- Explicit import sources before their contents are written to Notion.
- Explicit exports requested by a maintainer or user.
- Temporary files created and consumed inside one command execution.
- Failure diagnostics marked as non-canonical.

Local artifacts must not be used as canonical input for agent decisions when Notion is configured. If a local artifact contains useful business data, the system must import or sync it into Notion before using it in a real workflow.

## Canonical Read Model

### Output Generation Input

For real output generation, the input should be assembled from Notion:

- Opportunity page from `B2G Opportunities`.
- Related Target Account page from `B2G Target Accounts`.
- Related evidence pages from `B2G SECOP Research`.
- Relevant prior outputs from `B2G GTM Outputs`, when useful for continuity.

Local JSON input remains acceptable only for tests, previews, and import/migration workflows.
